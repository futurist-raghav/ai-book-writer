## Database Optimization

### Recommended Indexes

```sql
-- DeviceToken fast lookups by user
CREATE INDEX idx_device_tokens_user_id 
ON device_tokens(user_id) 
WHERE is_active = true;

-- Notification queries by user + read status
CREATE INDEX idx_notifications_user_read 
ON notifications(user_id, read) 
WHERE deleted_at IS NULL;

-- Notification list pagination
CREATE INDEX idx_notifications_created 
ON notifications(user_id, created_at DESC);

-- Token expiry cleanup
CREATE INDEX idx_device_tokens_updated 
ON device_tokens(updated_at DESC) 
WHERE is_active = false;

-- Preference lookups
CREATE INDEX idx_notification_prefs_user 
ON notification_preferences(user_id) 
UNIQUE;
```

### Analyze & Monitor

```sql
-- Monitor index usage
SELECT schemaname, tablename, indexname, idx_scan, idx_tup_read, idx_tup_fetch
FROM pg_stat_user_indexes
ORDER BY idx_scan DESC;

-- Check slow queries
SELECT query, calls, mean_time, max_time
FROM pg_stat_statements
WHERE query LIKE '%notification%'
ORDER BY mean_time DESC;
```

## Code Optimizations

### 1. Query Optimization - Notifications Service

**Before:**
```python
# N+1 query problem
notifications = db.query(Notification).filter(...).all()
for notif in notifications:
    pref = db.query(NotificationPreference).filter(...).first()
```

**After:**
```python
# Single query with eager loading
notifications = db.query(Notification).filter(...).all()
# Batch load preferences once
prefs = {p.user_id: p for p in db.query(NotificationPreference).all()}
```

### 2. Pagination - Optional Limit

**Before:**
```python
# Fetches all, then limits in Python
notifications = db.query(Notification).filter(...).all()[:50]
```

**After:**
```python
# Database-level pagination
notifications = db.query(Notification)\
    .filter(...)\
    .order_by(Notification.created_at.desc())\
    .limit(50)\
    .offset(offset)\
    .all()
```

### 3. Aggregation - Unread Count

**Before:**
```python
# Counts every row
unread = db.query(Notification).filter(
    Notification.user_id == user_id,
    Notification.read == False
).count()
```

**After:**
```python
# Use database aggregation
from sqlalchemy import func
unread = db.query(func.count(Notification.id))\
    .filter(
        Notification.user_id == user_id,
        Notification.read == False
    )\
    .scalar()
```

### 4. Bulk Operations - Mark as Read

**Before:**
```python
# Individual updates
for notif_id in notification_ids:
    notification = db.query(Notification).get(notif_id)
    notification.read = True
    db.commit()
```

**After:**
```python
# Bulk update
db.query(Notification)\
    .filter(Notification.id.in_(notification_ids))\
    .update({Notification.read: True})
db.commit()
```

### 5. Caching - User Preferences

**Before:**
```python
# Database hit on every send check
if should_notify(user_id, notification_type):
    send_notification(...)
```

**After:**
```python
# Cache with TTL
from functools import lru_cache
from datetime import timedelta

@cached(ttl=timedelta(minutes=5))
def get_user_preferences(user_id: str):
    return db.query(NotificationPreference).get(user_id)

if should_notify(get_user_preferences(user_id), notification_type):
    send_notification(...)
```

## Mobile Performance

### 1. Lazy Load Notification List

**Before:**
```typescript
const { notifications } = useNotifications();
// Loads entire list at once
```

**After:**
```typescript
const [limit, setLimit] = useState(50);

const { notifications } = useNotifications({
  limit: limit,
  onEndReached: () => setLimit(prev => prev + 50),
});
```

### 2. Optimize FlatList Rendering

**Before:**
```typescript
<FlatList
  data={notifications}
  renderItem={NotificationItemComponent}
  removeClippedSubviews={false}
/>
```

**After:**
```typescript
<FlatList
  data={notifications}
  renderItem={NotificationItemComponent}
  removeClippedSubviews={true}
  maxToRenderPerBatch={10}
  initialNumToRender={10}
  updateCellsBatchingPeriod={50}
  windowSize={5}
  keyExtractor={item => item.id}
/>
```

### 3. Memoize Components

**Before:**
```typescript
function NotificationItemComponent({ notification }) {
  // Re-renders on every parent update
  return <View>...</View>;
}
```

**After:**
```typescript
const NotificationItemComponent = React.memo(({ notification }) => {
  return <View>...</View>;
});
```

### 4. Debounce Mark as Read

**Before:**
```typescript
onPress={() => markAsRead(notification.id)} // Immediate update
```

**After:**
```typescript
const debouncedMarkAsRead = useCallback(
  debounce((id: string) => markAsRead(id), 500),
  []
);
```

## Network Optimization

### 1. Batch Device Tokens in Registration

**Before:**
```typescript
// One request per token generation
const token = await getToken();
await registerDevice(token);
```

**After:**
```typescript
// Register with metadata in single call
const token = await getToken();
await registerDevice(token, {
  platform: Platform.OS,
  app_version: Constants.buildVersion,
  timestamp: new Date(),
});
```

### 2. Compress Notification Data

**Before:**
```json
{
  "title": "Chapter Updated",
  "body": "Chapter 3 was updated by Alice",
  "data": {
    "chapter_id": "uuid-long-string",
    "chapter_title": "Chapter 3: The Beginning",
    "editor_name": "Alice",
    "editor_id": "uuid-long-string"
  }
}
```

**After:**
```json
{
  "title": "Chapter Updated",
  "body": "Chapter 3 was updated by Alice",
  "data": {
    "cid": "c123",
    "ct": "Ch3",
    "en": "Alice"
  }
}
```

### 3. Conditional API Calls

**Before:**
```typescript
// Always fetch on mount
useEffect(() => {
  fetchNotifications();
}, []);
```

**After:**
```typescript
// Only fetch if stale or empty
useEffect(() => {
  if (!notifications.length && !lastFetch) {
    fetchNotifications();
  } else if (Date.now() - lastFetch > 60000) {
    fetchNotifications();
  }
}, []);
```

## Scalability Strategy

### Current Scale
- **Devices:** ~1000s (single database)
- **Notifications/day:** ~10,000s
- **Concurrent sends:** ~100

### Scale to 10,000 users
1. **Database:** Add read replicas for notification queries
2. **Cache:** Redis for user preferences cache
3. **Message Queue:** Ensure Celery workers are autoscaling
4. **Batching:** Group sends by region for Expo API efficiency

### Scale to 100,000 users
1. **Sharding:** Partition notifications by user_id
2. **CDN:** Cache notification metadata
3. **Message Broker:** Switch to Kafka for high throughput
4. **Dedicated workers:** Notification sending tier separate from other tasks
