# Mobile Push Notifications Setup Guide

This guide explains how to set up Firebase Cloud Messaging (FCM) with Expo push notifications for the Scribe House mobile app.

## Architecture Overview

```
Mobile App (Expo)
    ↓ (Registers device)
    ↓
Backend API (/api/v1/notifications)
    ↓ (Stores device tokens)
    ↓
Backend Database (DeviceToken, Notification tables)
    ↓ (Notification sender)
    ↓
Expo Push Service (HTTPS)
    ↓ (Sends to FCM/APNs)
    ↓
Mobile App (Receives notification)
```

## Components

### Mobile Side

#### `/mobile/lib/notifications-service.ts`
Core notification management:
- `NotificationService` class: Handles permission requests, local + remote notifications
- `useNotifications()` hook: Subscribe to notification state
- `useNotificationTapped()` hook: Listen to notification tap events

#### `/mobile/lib/notifications-init.ts`
App initialization:
- `initializeNotifications()`: Called on app launch
- Requests permissions (iOS)
- Registers device token with backend
- Sets up notification channels (Android)

#### `/mobile/app/(main)/notifications/index.tsx`
Notification center UI:
- List all notifications
- Mark as read/unread
- Dismiss notifications
- Badge showing unread count in tab bar

### Backend Side

#### `/backend/app/models/notification_model.py`
Database models:
- `Notification`: Stores notification records
- `DeviceToken`: Tracks registered mobile devices
- `NotificationPreference`: User notification settings

#### `/backend/app/services/notification_service.py`
Business logic:
- Device token lifecycle (register, unregister, cleanup)
- Notification CRUD operations
- Read/unread tracking
- Preference management

#### `/backend/app/workers/push_notification_sender.py`
Notification sending:
- `PushNotificationSender` class: Sends to Expo via HTTPS
- `BulkNotificationSender` class: Batch sends to multiple users
- Event-specific helpers (chapter, collaboration, assignment, milestone)

#### `/backend/app/workers/notification_tasks.py`
Celery tasks (async background jobs):
- `send_push_notification`: Send notification in background
- `send_chapter_update_notification`: Chapter-specific
- `send_collaboration_notification`: Collaboration-specific
- `send_assignment_notification`: Assignment-specific
- `send_milestone_notification`: Milestone-specific
- `cleanup_inactive_devices`: Periodic cleanup (run daily)
- `refresh_device_token`: Handle token refresh
- `sync_notification_state`: Sync undelivered notifications

#### `/backend/app/api/routes/notifications.py`
REST API endpoints:
- `POST /register-device`: Register for push notifications
- `POST /unregister-device`: Unregister device
- `GET /`: List notifications
- `GET /{id}`: Get specific notification
- `PATCH /{id}/read`: Mark as read
- `POST /mark-all-read`: Mark all as read
- `DELETE /{id}`: Delete notification
- `GET /unread/count`: Get unread count
- `GET /preferences/me`: Get preferences
- `PATCH /preferences/me`: Update preferences
- `POST /send-test`: Send test notification

## Setup Instructions

### 1. Environment Variables

Add these to your `.env` files:

**Frontend/Mobile (.env.local or .env.prod):**
```env
EXPO_PUBLIC_FIREBASE_API_KEY=xxx
EXPO_PUBLIC_FIREBASE_PROJECT_ID=xxx
EXPO_PUBLIC_FIREBASE_MESSAGING_SENDER_ID=xxx
```

**Backend (.env):**
```env
# Notification settings
EXPO_PUSH_URL=https://exp.host/--/api/v2/push/send
NOTIFICATION_CLEANUP_ENABLED=true
NOTIFICATION_RETENTION_DAYS=30

# Celery (for background tasks)
CELERY_BROKER_URL=redis://redis:6379
CELERY_RESULT_BACKEND=redis://redis:6379
```

### 2. Mobile App Setup

**Step 1: Initialize in app layout** (`/mobile/app/_layout.tsx`)
```typescript
// Already added in _layout.tsx
import { initializeNotifications } from '../lib/notifications-init';

// In useEffect after auth hydration:
await initializeNotifications();
```

**Step 2: Register device on login** 
Auto-triggered by `initializeNotifications()` after auth

**Step 3: Add to navigation** (`/mobile/app/(main)/_layout.tsx`)
```typescript
// Already added
<Tabs.Screen
  name="notifications"
  options={{
    title: 'Notifications',
    tabBarIcon: ({ color, size }) => (
      <View>
        <Ionicons name="notifications" size={size} color={color} />
        <NotificationBadge />
      </View>
    ),
  }}
/>
```

### 3. Backend Setup

**Step 1: Create database tables**
```bash
# Run migrations (models should exist)
docker compose exec backend alembic upgrade head
```

**Step 2: Install dependencies**
```bash
# Add to backend/requirements.txt if not already there:
aiohttp>=3.8.0
expo-server-sdk>=0.4.0  # Optional: if using SDK
```

**Step 3: Configure Celery tasks** (if queue workers not already running)
```bash
# In docker-compose.yml or your process manager:
celery_worker:
  command: celery -A app.celery_worker worker --loglevel=info
```

**Step 4: Set up notification cleanup cron job**
```bash
# Add to celery beat schedule in settings:
CELERY_BEAT_SCHEDULE = {
    'cleanup-inactive-devices': {
        'task': 'app.workers.notification_tasks.cleanup_inactive_devices',
        'schedule': crontab(hour=2, minute=0),  # Daily at 2 AM
    },
}
```

### 4. Firebase Setup (Optional - for FCM specifics)

If you want deeper FCM integration:

1. Create Firebase project at https://console.firebase.google.com
2. Add Android and iOS apps
3. Download service account key (JSON)
4. Configure in Expo:
   ```bash
   eas secret:create FIREBASE_SERVICE_ACCOUNT_KEY
   # Paste JSON content
   ```
5. Update `eas.json` to include secret in build

**Note:** For Expo, FCM is handled transparently through Expo's push service.

## Testing

### Test via API

**1. Register a device (from mobile app automatically, or manual test):**
```bash
curl -X POST http://localhost:8000/api/v1/notifications/register-device \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "device_token": "ExponentPushToken[xxx]",
    "platform": "ios",
    "app_version": "1.0.0"
  }'
```

**2. Send test notification:**
```bash
curl -X POST http://localhost:8000/api/v1/notifications/send-test \
  -H "Authorization: Bearer YOUR_TOKEN"
```

**3. Check notifications:**
```bash
curl http://localhost:8000/api/v1/notifications \
  -H "Authorization: Bearer YOUR_TOKEN"
```

### Test via Mobile App

1. Open app (triggers automatic device registration)
2. Go to Settings → Notifications
3. Toggle notification types (auto-saves preferences)
4. Have another user send a collaboration invitation
5. Verify notification appears in Notifications tab

### Test Undelivered Notifications

1. Close app while notification is being sent
2. Reopen app
3. Run `sync_notification_state` task
4. Notifications should appear

## Troubleshooting

### Notifications Not Received

1. **Check device is registered:**
   ```bash
   curl http://localhost:8000/api/v1/notifications/unread/count \
     -H "Authorization: Bearer YOUR_TOKEN"
   ```

2. **Check notification preferences:**
   ```bash
   curl http://localhost:8000/api/v1/notifications/preferences/me \
     -H "Authorization: Bearer YOUR_TOKEN"
   ```

3. **Check device tokens in DB:**
   ```sql
   SELECT * FROM device_tokens WHERE user_id = 'YOUR_USER_ID';
   ```

4. **Check notification logs:**
   ```bash
   docker compose logs backend | grep notification
   ```

### Token Registration Fails

1. Check mobile app has internet connection
2. Verify `EXPO_PUBLIC_API_URL` points to correct backend
3. Check backend is running and `/api/v1/notifications/register-device` endpoint is accessible
4. Check auth token is valid

### High Memory Usage from Notifications

1. Enable `NOTIFICATION_CLEANUP_ENABLED=true`
2. Reduce `NOTIFICATION_RETENTION_DAYS` (default 30)
3. Run cleanup task: `docker compose exec backend celery -A app.workers.notification_tasks cleanup_inactive_devices`

## Performance Considerations

1. **Device token cleanup:** Runs daily, removes inactive tokens older than 30 days
2. **Query optimization:** Add indexes to `device_tokens.user_id` and `notifications.user_id`
3. **Batch sending:** Use `BulkNotificationSender` for events affecting multiple users
4. **Rate limiting:** Consider adding rate limits to `/send-test` endpoint in production

## Security Notes

1. **Token validation:** All endpoints protected by auth guards (`get_current_user`)
2. **User isolation:** Devices belong to users, notifications are user-scoped
3. **Preference enforcement:** Respects user notification settings
4. **HTTPS only:** Expo push service requires HTTPS
5. **Token refresh:** Implement token refresh on app update to handle Expo token changes
